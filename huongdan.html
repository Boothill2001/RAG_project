1 RAG là gì? (giải thích như cho người chưa biết gì)

Tưởng tượng ChatGPT là một người rất thông minh nhưng trí nhớ đóng băng tại thời điểm nó được train. Hỏi nó về tài liệu nội bộ công ty bạn → nó không biết → nó bịa (hallucination).

RAG (Retrieval-Augmented Generation) giải quyết bằng cách chia làm 2 bước:

1️⃣ Retrieval (tìm kiếm) — khi user hỏi, hệ thống đi tìm những đoạn tài liệu liên quan nhất trong kho tri thức (knowledge base) của bạn.

2️⃣ Generation (sinh câu trả lời) — đưa các đoạn đó vào prompt và ra lệnh cho LLM: "Chỉ được trả lời dựa trên các đoạn này, kèm trích nguồn, không biết thì nói không biết."

💡 Kết quả: câu trả lời chính xác, có nguồn, cập nhật được mà không cần train lại model. Đây là kiến trúc phổ biến nhất trong AI doanh nghiệp — và là thứ nhà tuyển dụng AI Engineer hỏi nhiều nhất.

2 Project này "xịn" hơn RAG cơ bản ở đâu?

RAG tutorial trên mạng chỉ có: embed câu hỏi → tìm vector gần nhất → đưa vào LLM. Project này làm 4 nâng cấp giống hệ thống production thật — khớp từng dòng trong CV:

✂️

Semantic Chunking

Cắt tài liệu theo heading / đoạn văn (ranh giới ngữ nghĩa) thay vì cắt cứng mỗi 500 ký tự → mỗi chunk nói đúng 1 chủ đề → embedding "sắc nét" hơn.

🔀

Hybrid Search

Chạy song song 2 kiểu tìm: dense (FAISS — bắt câu cùng ý khác từ) và sparse (BM25 — bắt từ khóa chính xác, mã lỗi, tên riêng). Trộn kết quả bằng RRF: cộng điểm 1/(60+hạng).

🎯

Re-ranking

Lấy ~20 ứng viên, cho qua cross-encoder — model đọc cả câu hỏi lẫn đoạn văn cùng lúc nên chấm điểm chính xác hơn nhiều → chọn top 5 thật sự liên quan.

🛡️

Graceful Fallback

Không có key / LLM sập / hết tiền → hệ thống không chết, trả về chế độ extractive: trích nguyên văn đoạn liên quan nhất. Demo phỏng vấn không bao giờ đứng hình.

📏 Ngoài ra còn có script đo lường (evaluation) chứng minh bằng con số rằng hybrid tốt hơn dense-only — kể trong phỏng vấn cực kỳ ăn điểm (xem mục 7).

5 Cài đặt từng bước (Windows)

⚠️ Yêu cầu: Python 3.10+ — tải tại python.org, nhớ tick "Add to PATH" khi cài.

Mở PowerShell tại thư mục project, chạy lần lượt:

  1. Tạo môi trường ảo — cái "hộp" chứa thư viện riêng cho project, không đụng Python hệ thống. 
    python -m venv .venv
  2. Kích hoạt môi trường ảo — dòng lệnh sẽ hiện (.venv) ở đầu. 
    .venv\Scripts\activate
  3. Cài thư viện — lần đầu tải ~500MB (có PyTorch), chờ vài phút. 
    pip install -r requirements.txt
  4. Cấu hình API key — không bắt buộc: không có key vẫn chạy được ở chế độ fallback. 
    copy .env.example .env
# Mở .env bằng Notepad, thay your_deepseek_api_key_here bằng key thật
# Lấy key tại platform.deepseek.com — nạp $2 là dùng thoải mái
  5. Build index — "nạp kiến thức" — lần đầu tải model embedding ~90MB. 
    python scripts/build_index.py
# Thấy "Done in X.Xs. Index files written to ...\index" là OK ✅

9 Câu hỏi phỏng vấn thường gặp (bấm để mở)

Tại sao dùng hybrid mà không dùng dense-only?

Embedding yếu với token lạ (mã sản phẩm, mã lỗi, viết tắt) vì model chưa từng học; BM25 yếu khi user diễn đạt khác từ ngữ trong tài liệu. Chạy cả hai rồi fuse bằng RRF lấy được điểm mạnh của cả hai — benchmark doanh nghiệp thường tăng recall 10–25%.

RRF là gì, sao không cộng thẳng điểm?

Điểm BM25 (0→∞) và cosine (−1→1) khác thang đo — cộng thẳng phải normalize và tune trọng số. RRF chỉ dùng thứ hạng: score = Σ 1/(60 + rank) — miễn nhiễm khác thang đo, không cần tune, là chuẩn de-facto trong production.

Bi-encoder vs cross-encoder khác gì nhau?

Bi-encoder embed câu hỏi và tài liệu riêng rẽ rồi so cosine — nhanh (precompute được) nhưng hai bên không "nhìn thấy nhau". Cross-encoder ghép cả hai vào một input, mọi token attend lẫn nhau — chính xác hơn hẳn nhưng chậm, nên chỉ dùng re-rank ~20 ứng viên cuối.

Làm sao đo chất lượng một hệ RAG?

Tách 2 tầng: retrieval đo hit-rate@k / MRR trên bộ câu hỏi có gán nguồn đúng; generation đo faithfulness / answer relevance bằng LLM-as-judge (RAGAS, TruLens). Project này có scripts/evaluate.py làm ablation retrieval.

Đưa lên production thì cần thêm gì?

Thay FAISS bằng vector DB có filtering + replication (Qdrant/Pinecone); thêm caching (embedding, answer); auth + rate limiting; Docker + K8s + CI/CD; observability (trace từng stage, cost tracking); index rebuild theo blue-green; chống prompt injection từ tài liệu.

Sao lại chọn DeepSeek?

API tương thích chuẩn OpenAI (đổi provider chỉ 1 dòng config), rẻ, chất lượng tốt. Kiến trúc không khóa vào provider nào — swap sang Claude/GPT chỉ cần sửa config.py.

Semantic chunking hơn gì cắt cứng theo ký tự?

Cắt cứng thường chém đôi câu văn, bảng, code block → chunk vô nghĩa, embedding nhiễu. Cắt theo heading/đoạn văn giữ mỗi chunk về đúng 1 chủ đề → embedding sắc nét, retrieval chính xác hơn. Vẫn cần cap kích thước tối đa + overlap để không mất ngữ cảnh ở ranh giới.
