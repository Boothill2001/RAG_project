# Evaluating RAG Systems

## Why evaluation is hard

A RAG system has two coupled components — retrieval and generation — and an error in
either produces a bad answer. Good evaluation measures them separately: retrieval
metrics tell you whether the right evidence reached the prompt; generation metrics tell
you whether the model used that evidence correctly.

## Retrieval metrics

- **Hit rate @ k (recall @ k)**: fraction of questions for which at least one truly
  relevant chunk appears in the top-k retrieved results. The single most important
  retrieval health metric — if the evidence never reaches the prompt, generation
  cannot succeed.
- **MRR (Mean Reciprocal Rank)**: average of 1/rank of the first relevant result.
  Rewards putting the right chunk near the top.
- **nDCG**: rank-weighted gain that handles graded relevance judgments.
- **Precision @ k**: fraction of retrieved chunks that are relevant; matters because
  irrelevant chunks waste prompt tokens and distract the model.

## Building a retrieval test set

Create question-to-source pairs: for each test question, record which document (or
chunk) contains the answer. Even 20-50 hand-written pairs catch most regressions.
Synthetic test sets can be generated by asking an LLM to write questions for each
chunk, then verifying the chunk answers its own question.

## Generation metrics

- **Faithfulness / groundedness**: every claim in the answer must be supported by the
  retrieved context. Measured by an LLM judge decomposing the answer into claims and
  verifying each against the context.
- **Answer relevance**: does the answer actually address the question?
- **Correctness**: agreement with a gold reference answer, when one exists.

Frameworks such as RAGAS, TruLens, and DeepEval implement these metrics with
LLM-as-judge pipelines.

## Ablation testing

The most convincing way to justify pipeline complexity is ablation: run the same test
set through dense-only retrieval, BM25-only, hybrid, and hybrid + re-ranking, and
compare hit rates. Typical enterprise results show hybrid beating either single
retriever, and re-ranking adding a further boost to top-3 precision. Report the numbers
in your README — concrete metrics build far more trust than adjectives.

## Online monitoring

In production, log every query with its retrieved chunks, scores, latency per stage,
token usage, and user feedback (thumbs up/down). Alert on retrieval score drift, rising
"I don't know" rates, and latency regressions. Store traces so bad answers can be
replayed and debugged offline.
